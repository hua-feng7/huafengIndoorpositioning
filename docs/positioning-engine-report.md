# PositioningEngine 算法检查与修正报告

## 1. 检查目标

本次检查聚焦于 `PositioningEngine.kt` 中的定位主流程，重点关注以下几个问题：

- 距离分数是否具有可比性
- 聚类预筛选是否会破坏防抖逻辑
- 时间平滑是否混入了算法核心，导致结果语义不一致

相关文件：

- `app/src/main/java/com/huafeng/beaconzone/PositioningEngine.kt`
- `app/src/main/java/com/huafeng/beaconzone/MapCollectScreen.kt`

## 2. 原实现中存在的问题

### 问题一：距离分数未归一化

原先距离计算方式是：

- 对 `liveRssi` 与 `fingerprint.rssiMap` 的 key 并集逐项求差
- 有值缺失时直接使用固定罚分 `missingPenalty`
- 最后返回 `sqrt(sum)`

这个写法的主要问题是：

- 特征越多，总和天然越大
- 记录了更多 Beacon / WiFi 的指纹点，容易因为“维度更多”而被额外惩罚
- 在 `BLE+WiFi` 模式下尤为明显，因为两种信号源混合后特征数会明显增加

结果就是：

- 分数不再是“平均意义上的差异”
- 不同指纹点之间的距离值不完全可比
- 算法会偏向特征更少、信号更稀疏的指纹点

### 问题二：聚类预筛选可能绕过防抖

原先流程是：

1. 先通过 `prefilterFingerprintsByCluster()` 只保留部分候选簇
2. 再在候选簇内算距离和做 hysteresis

问题在于：

- `applyHysteresis()` 依赖 `currentFingerprintId`
- 但如果当前点所在簇在预筛选阶段被过滤掉，那么它根本不会出现在 `scored` 列表里
- 一旦当前点不在 `scored` 中，防抖逻辑就无法和“当前点”做直接比较

结果就是：

- 参数上虽然设置了防抖阈值
- 实际上只要聚类预筛选先把当前簇排除，防抖就会失效
- 会出现“明明开了防抖，但还是突然跳点”的现象

### 问题三：时间平滑混入算法核心，结果语义不一致

原先 `PositioningEngine` 在返回结果前直接对 `PositionEstimate` 做时间平滑：

- `xPx / yPx` 被平滑
- 但 `fingerprintId / distance` 仍然保留新候选点的值

这会导致一个语义错位：

- 坐标可能已经是“上一次”和“这一次”之间的中间位置
- 但 `fingerprintId` 却已经切到新点
- `distance` 也是新点距离

结果就是：

- 坐标与身份不一致
- 如果上层继续把 `fingerprintId` 当成当前点参与防抖，会带来额外理解成本
- 严格来说，这不应该属于算法核心输出，而应该是显示层的后处理

## 3. 本次修正内容

### 修正一：距离改为归一化 RMSE 风格分数

现在距离函数改为：

- 仍然对 key 并集逐项比较
- 缺失项仍然使用 `missingPenalty`
- 但最终返回 `sqrt(sum / keys.size)`

这样做的好处：

- 分数变成“每个特征维度上的平均误差”
- 特征数不同的指纹点之间，距离更可比
- 混合 `BLE+WiFi` 场景下更稳定

### 修正二：预筛选后强制保留当前点

现在在聚类预筛选完成后，会额外执行一次：

- 如果 `currentFingerprintId` 不在候选列表中
- 就从完整指纹集里把这个当前点补回候选列表

这样做的结果：

- 即使当前点所在簇在预筛选中被排除
- `applyHysteresis()` 仍然能和当前点做直接对比
- 防抖参数不会再被聚类预筛选“偷绕过去”

### 修正三：时间平滑从算法核心移到显示层

现在 `PositioningEngine` 只负责：

- 选出当前匹配点
- 输出当前匹配点的真实 `fingerprintId`
- 输出当前匹配点的原始估计坐标和距离

时间平滑改为在 `MapCollectScreen.kt` 中处理，只用于红点显示：

- 算法核心结果保持“身份一致”
- 红点渲染时才做指数平滑
- `fingerprintId` 与 `distance` 始终对应真实匹配点

这样做的好处：

- 算法输出更干净
- 结果语义更一致
- 后续如果要做调试日志、候选点分析、轨迹回放，会更容易解释

## 4. 修正后的当前逻辑

当前定位主流程可以概括为：

1. 构建实时 RSSI 向量
2. 按算法需要进行最强 N 信号筛选
3. 按聚类预筛选缩小候选集
4. 强制补回当前点，保证防抖有效
5. 对候选点计算归一化距离
6. 通过 1NN / KNN / WKNN 等算法得到候选位置
7. 对候选结果应用防抖
8. 在 UI 层对显示红点做可选时间平滑

## 5. 这次修正带来的直接效果

### 对混合信号定位

- `BLE+WiFi` 下分数可比性更好
- 不容易再系统性偏向“特征更少”的点

### 对防抖逻辑

- 聚类预筛选不再偷偷绕过防抖
- `switchMargin` 的行为更符合参数预期

### 对显示与调试

- 红点依然可以平滑
- 但算法内部输出保持纯净
- 后续要分析“到底匹配到了哪个参考点”会更清楚

## 6. 本轮继续修正：BLE / WiFi 分开评分再融合

在上一轮结构修正完成后，又继续实现了“按信号源分开评分，再做融合”的逻辑。

### 原先的问题

虽然距离已经改成了归一化 RMSE 风格分数，但 BLE 与 WiFi 之前仍然是混在同一个并集中一起计算的。

这会带来两个直接问题：

- `BLE_ONLY` 模式下，如果指纹点里还保留着 WiFi 数据，会被 WiFi 项额外拖分
- `WIFI_ONLY` 模式下，同理也会被 Beacon 项影响

也就是说，实时扫描端虽然已经按模式切换了数据源，但距离函数还没有真正做到“只比较当前模式应该参与的那一类信号”。

### 这次的实现方式

现在距离计算改成：

1. 先根据 `liveRssi` 中实际出现的 key 判断当前活跃信号族
2. 把 `BLE` 和 `WiFi` 分开计算各自的归一化距离
3. 再对各个信号族的距离做融合平均

现在的行为变成：

- `仅BLE` 时只看 BLE 距离
- `仅WIFI` 时只看 WiFi 距离
- `BLE+WIFI` 时分别打分后再融合

### 这次修正带来的效果

- 模式切换语义终于和距离计算一致了
- 指纹点中“另一种信号源”的历史数据不会再干扰当前模式
- 混合定位时 BLE 与 WiFi 的作用边界更清晰

## 7. 本轮继续优化：仅 BLE 模式独立计算并提高精度

在完成“BLE / WiFi 分开评分再融合”之后，又继续把 BLE 本身单独做了专门优化，目标是优先提高 `仅BLE` 模式下的定位精度。

### 这次为什么还要继续改

虽然前一轮已经把 BLE 和 WiFi 拆开了，但 BLE 之前仍然只是沿用“通用距离模板”：

- 共同 key 就直接比较 RSSI 差值
- 缺失项就加固定罚分

这对 BLE 来说还不够好，因为 Beacon 指纹有两个很明显的特点：

- 强信号 Beacon 通常更有区分力
- 共同命中的 Beacon 数量，对定位可信度影响很大

如果仍然把 BLE 当成普通通用向量处理，就会丢掉这两个信息。

### 这次具体改了什么

现在 BLE 走的是独立距离函数，不再和 WiFi 共用同一套细节：

1. `仅BLE` 模式下，距离只看 BLE 信号族
2. BLE 共同命中的 Beacon 会按信号强度加权
3. 共同命中比例越低，额外缺失惩罚越高
4. 如果一个共同 Beacon 都没有，直接判成高距离

也就是说，BLE 现在不再只是“差值平方和”，而是：

- 共同 Beacon 的加权 RMSE
- 再加上基于重叠率的缺失惩罚

### 这次修正带来的效果

- `仅BLE` 模式彻底和 WiFi 解耦
- 强 Beacon 对定位的贡献更大
- Beacon 重叠率不足时，会更谨慎地拉开距离
- 对 Beacon 稀疏、遮挡、偶发丢失的场景更鲁棒

## 8. 当前仍然存在的限制

这次修正解决的是结构性问题，但还不是最终形态，仍有几个后续可优化点：

### 1. 缺失惩罚仍然是统一常量

当前 `missingPenalty` 仍然对：

- BLE
- WiFi
- 强信号
- 弱信号

一视同仁。

这意味着：

- 不同信号源的重要性还没有区分
- WiFi 与 BLE 混合时仍有进一步优化空间

### 2. 聚类方式仍然是规则网格

当前聚类预筛选是：

- 按坐标网格切桶
- 再用每个桶的平均指纹做粗筛选

它的优点是简单稳定，但不是严格意义上的聚类算法。后续如果需要，可以升级为：

- 距离阈值聚类
- KMeans
- 先按房间/区域分组再做簇筛选

### 3. 距离函数还没有做信号源独立权重融合

当前已经做到 BLE / WiFi 分开评分再融合，但还没有做到“分源独立权重配置”。更进一步的优化可以是：

- 给 BLE 与 WiFi 各自设置融合权重
- 或按当前有效信号数量、置信度做动态融合

这样会更适合混合定位。

## 9. 推荐的下一步

如果继续沿着这条线优化，我建议优先顺序是：

1. 给 BLE 增加“仅使用前 N 个稳定 Beacon”的可调模式
2. 给 `missingPenalty` 增加按 Beacon 稳定度区分的权重
3. 给聚类预筛选增加“保留当前簇优先”的更强约束
4. 在调试页显示候选点前 3 名及其归一化距离

## 10. 总结

到当前这一版为止，主要做了五件事：

- 把距离从“累加总误差”改成“归一化平均误差”
- 保证聚类预筛选不会破坏防抖
- 把时间平滑从算法核心移到显示层
- 把 BLE / WiFi 改成分开评分再融合
- 给 BLE 增加独立的加权距离与重叠率惩罚

整体效果是：

- 算法分数更合理
- 参数行为更符合直觉
- 模式切换与距离计算逻辑一致
- 仅 BLE 模式下的定位逻辑更独立、更贴近 Beacon 指纹特性
- 结果解释性更强

这版已经比之前更适合作为后续继续扩展算法的基础版本。
